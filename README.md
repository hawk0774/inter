## Образы/Контейнеры

А как устроен образ?

<details>
  <summary>Ответ</summary>
Манифест Docker — это документ в формате JSON, который описывает образ Docker, указывая его слои, размер и операционную систему/архитектуру, для которой он был создан. В основном он используется для управления образами для разных архитектур, что позволяет использовать один тег образа на разных платформах (например, linux/amd64, linux/arm64).

Конфигурация - Файл конфигурации в образе определяет свойства контейнера, такие как переменные среды, команда по умолчанию (CMD) и точка входа (ENTRYPOINT), при запуске. 

Слои в Docker в Linux организуются с помощью драйверов хранилища (обычно overlay2), которые объединяют несколько неизменяемых слоев образа (readonly) и один записываемый слой контейнера (read-write) в единую файловую систему. Каждый слой — это результат выполнения команды RUN, COPY или ADD в Dockerfile, что позволяет экономить место и кешировать сборку. 
Основные принципы организации слоев: 
Иерархия (Union FS): Слои накладываются друг на друга. Слой N включает только изменения (добавленные, измененные или удаленные файлы) по сравнению со слоем N-1
Только чтение (Read-Only): Слои образа неизменяемы. При обновлении файла он копируется с нижнего слоя на верхний записываемый слой (механизм Copy-on-Write)
Записываемый слой (Read-Write): При запуске контейнера поверх слоев образа добавляется временный тонкий слой для записи данных.
Кеширование: Docker кеширует каждый слой. Если инструкция Dockerfile не менялась, используется готовый слой, что ускоряет сборку.
Хранение: Все слои хранятся в /var/lib/docker/overlay2/
</details>

```
ARG UBUNTU_VERSION="latest"
# ГЛОБАЛЬНЫЙ ARG: Объявлен до FROM, а значит, будет недоступен внутри этапов сборки ниже без повторного объявления.

FROM ubuntu:${UBUNTU_VERSION} AS builder

RUN echo "Starting build ${BUILD_VERSION}"

# АНТИПАТТЕРН: Копирование всего контекста сразу (включая .git, логи и локальные конфиги) инвалидирует кэш при любом изменении файла.
COPY . /app
WORKDIR /app

# АНТИПАТТЕРН: Раздельные вызовы apt-get. Если закэшируется update, но изменятся зависимости в install, сборка упадет или поставит старье.

RUN pip install -r requirements.txt

# ОШИБКА: Запуск тестов внутри сборки. Dockerfile — для сборки артефакта, а не для CI. Если тесты требуют БД, билд упадет.
RUN pytest tests/ 

# АНТИПАТТЕРН: Смешивание разных экосистем (Python и Node.js) в одном этапе раздувает слой.
RUN npm install
RUN npm run build 

# --- ЭТАП 2: RUNNER (Финальный образ) ---
FROM ubuntu:${UBUNTU_VERSION}

# ОШИБКА SCOPE: Метка будет пустой (version=""), так как ARG BUILD_VERSION не объявлен в этом этапе.
LABEL version=${BUILD_VERSION}

COPY --from=builder / /

ENV DATABASE_PASSWORD="super_secret_password_123"
ENV API_KEY="AIzaSyA-example-key"

# АНТИПАТТЕРН: ADD автоматически распаковывает архивы. Если нужно просто скопировать файл — лучше COPY. Если архив битый — билд упадет неявно.
ADD my_app_data.tar.gz /data

# АНТИПАТТЕРН: Установка SSH-сервера. В контейнер нужно заходить через 'docker exec', а не через SSH. Это лишний вектор атаки и лишний вес.
RUN apt-get update && apt-get install -y openssh-server

WORKDIR /app
# АНТИПАТТЕРН: Запуск от имени root (по умолчанию). Если приложение взломают, у злоумышленника будет полный доступ к контейнеру.

# 1. Если db_migrator упадет, main.py всё равно попытается запуститься.
# 2. Использование '&' (background) приведет к тому, что Docker не будет следить за процессом main.py.
# 3. PID 1 станет nginx, и если python-приложение упадет, контейнер будет продолжать "работать" (zombie state).
CMD python3 db_migrator.py && python3 main.py & nginx -g "daemon off;"
```

Ты запускаешь Docker-контейнер с кастомным скриптом в качестве ENTRYPOINT:

```
COPY my_script.sh /
ENTRYPOINT ["/my_script.sh"]
```

Внутри скрипта my_script.sh запускается какой-то бинарный процесс (например, сервер), но без использования команды exec.

Когда ты выполняешь команду docker stop <container>, Docker посылает сигнал SIGTERM. Однако твое приложение игнорирует его и продолжает работать, пока Docker через 10 секунд не убивает контейнер жестким SIGKILL.
Почему так происходит, и как исправить ситуацию двумя разными способами (на уровне Dockerfile и на уровне Bash-скрипта)?

<details>
    <summary>Ответ</summary>

PID 1 и ядро: В Linux процесс с ID 1 (init) особенный. Ядро не применяет к нему действия по умолчанию для сигналов (например, завершение при SIGTERM). Если процесс специально не прописал обработчик (handler) для SIGTERM, он его просто проигнорирует.
Bash-скрипт как посредник: Если в Dockerfile прописано ENTRYPOINT ["/my_script.sh"], то именно Bash становится PID 1.
Отсутствие трансляции: Bash по умолчанию не пересылает сигналы своим дочерним процессам. Docker шлет SIGTERM скрипту, скрипт его игнорирует (или просто ждет завершения дочернего процесса), а само приложение внутри даже не знает, что его хотят выключить.

Способ 1: Использование exec (Уровень Bash)

Самый правильный и «чистый» путь. Измени запуск приложения внутри скрипта:
```
# Было:
./my_app

# Стало:
exec ./my_app
```

Команда exec не создает новый процесс, а замещает текущий процесс (Bash) твоим приложением. Твоё приложение само становится PID 1 и напрямую получает SIGTERM от Docker.

Способ 2: Использование Tini (Уровень Dockerfile)
Если нельзя менять скрипт или приложение не умеет корректно работать как PID 1 (например, не «жнет» зомби-процессы), используй легковесный инициализатор.
В современных версиях Docker это делается флагом:

```
docker run --init ...
```

Или через Dockerfile

```
# Устанавливаем tini
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--", "/my_script.sh"]
```

</details>

А вот контейнер, как он выглядит в Linux? С помощью каких инструментов происходит изоляция?

<details>
    <summary>Ответ</summary>
Контейнер — это просто обычный процесс, запущенный с помощью функций ядра clone() и unshare(), который ограничен лимитами и видит только то, что ему разрешили.

В основе работы контейнеров в Linux лежат три «кита» ядра, которые позволяют изолировать процессы друг от друга, не запуская при этом полноценную виртуальную машину.

Namespaces (Пространства имен)
Это главный механизм изоляции. Он заставляет процесс думать, что у него есть собственные ресурсы, отделенные от остальной системы.
PID: процесс видит себя как PID 1, а другие процессы в системе ему не видны.
NET: свой сетевой стек (свои IP, порты, маршруты).
MNT: своя файловая система (контейнер не видит файлы хоста).
UTS: собственное имя хоста (hostname).
IPC: изоляция межпроцессного взаимодействия.

Control Groups (cgroups)
Если namespaces изолируют обзор, то cgroups ограничивают ресурсы. Они следят, чтобы контейнер не «съел» всё на сервере:
Лимиты по CPU (например, не более 0.5 ядра).
Лимиты по RAM (например, 512 МБ).
Приоритеты ввода-вывода (Disk I/O).

Union File System (UnionFS)
Технология, которая позволяет «склеивать» разные слои образа в единую файловую систему.
Именно благодаря ей слои образа доступны только для чтения, а все изменения пишутся в верхний временный слой (Copy-on-Write).
Самые популярные реализации: Overlay2 (стандарт в Docker), AUFS.

Capabilities (Возможности)
Это способ разделить права root на мелкие части. Вместо того чтобы давать контейнеру полный доступ суперпользователя, Linux позволяет выдать только нужные привилегии (например, только право менять время или управлять сетевыми интерфейсами).
</details>

## Оркестрация (Kubernetes)

Расскажи про архитектуру k8s

<details>
    <summary>Ответ</summary>
На концептуальном уровне компоненты Kubernetes сгруппированы в два определённых класса: компоненты мастера (Master) и компоненты воркера (Worker).

Мастеры отвечают за управление всем кроме исполнения подов. Компоненты мастера включают в себя:

kube-controller-manager
kube-scheduler
kube-apiserver
etcd

Воркеры отвечают за управление исполнением подов. У них один компонент:

kubelet

Воркеры тривиально надёжны: временный или постоянный отказ любого воркера в кластере не затрагивает мастера или других воркеров кластера. Если приложение развёрнуто соответствующим образом, Kubernetes (пере-)планирует и (пере-)запускает любой под, затронутый отказом воркера.
</details>

При инициализации кластера кто-то создал под, но в этот момент сам спи сервер ещё не был запущен. Кто создал под?
<details>
    <summary>Ответ</summary>
Ситуация кажется парадоксальной: как можно отправить запрос на создание ресурса, если «мозг» кластера (kube-apiserver) ещё не работает?
В Kubernetes существует специальный механизм именно для таких случаев. Скорее всего, этот под создал Kubelet через механизм Static Pods (Статичные поды).

Как это работает?
Когда вы инициализируете кластер (например, через kubeadm), компоненты управления (Control Plane) сами должны где-то запуститься. Но они не могут быть запущены обычным способом через API, так как API еще не существует.
Локальный манифест: На узле (master-ноде) в определенной директории (обычно /etc/kubernetes/manifests/) лежат YAML-файлы.
Роль Kubelet: Агент kubelet запускается как системная служба (systemd). Он начинает сканировать эту директорию еще до того, как установит связь с API-сервером.
Самостоятельный запуск: Видя YAML-файл, kubelet самостоятельно запускает под (например, сам kube-apiserver, etcd или kube-scheduler).
</details>

Сколько контейнеров может быть в поде и какой есть точно?
<details>
    <summary>Ответ</summary>

В Kubernetes под (Pod) — это минимальная единица развертывания, и количество контейнеров в нем может варьироваться.

Сколько контейнеров может быть в поде?
Минимум: 1 основной контейнер. Под не может существовать без запущенных процессов.
Максимум: Технического жесткого лимита в коде Kubernetes нет, но на практике редко запускают более 3–5 контейнеров. Большое количество контейнеров в одном поде затрудняет масштабирование, управление ресурсами и отладку.



Какой контейнер есть точно (скрытый)?
Даже если вы описываете в манифесте всего один контейнер, внутри каждого пода всегда создается так называемый Pause Container (инфраструктурный контейнер).
Зачем он нужен?
Удержание пространств имен (Namespaces): Он запускается первым и «держит» сетевой стек (Network Namespace) и другие ресурсы (IPC).
Общий IP: Все остальные контейнеры пода, которые вы добавили сами, подключаются к сети этого Pause-контейнера. Именно поэтому они могут обращаться друг к другу через localhost.
Управление процессами: Он выступает в роли PID 1 для пода и «убирает» процессы-зомби.


Типы контейнеров, которые вы можете добавить:
Помимо основного приложения, в поде часто встречаются:
Sidecar Containers: Вспомогательные контейнеры, которые работают параллельно с основным (например, Envoy в Service Mesh или Vault Agent для проброса секретов).
Init Containers: Запускаются до основного контейнера и завершаются. Нужны для подготовки среды (например, миграция базы данных или ожидание доступности другого сервиса).
Ephemeral Containers: Временные контейнеры, которые можно запустить в уже работающем поде для отладки (kubectl debug).
</details>

Опишите жизненный цикл создания Пода. На каком этапе и как именно задействуется CNI?

<details>
    <summary>Ответ</summary>
Жизненный цикл создания Пода (Pod Creation Workflow) — это классический вопрос для проверки понимания архитектуры Kubernetes. Роль CNI здесь критическая, но она вступает в игру не сразу.

Вот основные этапы, где задействован CNI:
1. Запись в etcd (Control Plane)
Сначала пользователь отправляет манифест через kubectl. API Server проверяет его, сохраняет в etcd, а Scheduler назначает Поду конкретную ноду. На этом этапе про сеть еще никто не «думает».

2. Подхват задачи узлом (Kubelet)
Kubelet на целевой ноде замечает, что ему назначен новый Под. Он дает команду Container Runtime (например, containerd или Docker) создать «песочницу» (Pod Sandbox).

3. Создание Sandbox-контейнера (Pause container)
Runtime создает инфраструктурный контейнер (обычно это pause), который будет держать сетевое пространство имен (Network Namespace). Пока у этого контейнера есть только петлевой интерфейс (lo), своего IP в кластерной сети еще нет.

4. ВЫЗОВ CNI (Критический этап) ⚡
Как только сетевое пространство имен создано, Container Runtime (а не сам Kubelet напрямую!) обращается к CNI-плагину.
Что делает Runtime: Он ищет исполняемые файлы плагина в /opt/cni/bin и конфиг в /etc/cni/net.d.
Команда: Вызывается метод ADD.
Действия CNI-плагина:
Создает виртуальную пару интерфейсов (veth-пару).
Один конец помещает в сетевое пространство имен Пода (переименовывает в eth0).
Второй конец подключает к мосту (bridge) на хосте или настраивает маршрутизацию (зависит от типа CNI: Flannel, Calico и т.д.).
Выделяет IP-адрес через IPAM-драйвер (Host-local или DHCP).
Настраивает маршруты (Routes) внутри Пода.

5. Ответ и запуск приложений
CNI возвращает Runtime'у JSON с данными о выделенном IP и настройках. Только после этого Runtime начинает скачивать образы и запускать основные контейнеры вашего приложения. Все они будут использовать ту же сетевую среду, которую CNI подготовил для pause-контейнера.


CRI (Container Runtime Interface) — это основной способ связи между компонентом kubelet в кластере Kubernetes и исполняемой средой контейнеров. С помощью CRI Kubernetes взаимодействует с контейнерным движком который создает контейнеры (сам Kubernetes не создает контейнеры а только управляет ими). В качество CRI можно использовать такие движки как containerd, CRI-O.
</details>

Допустим, у нас развернуто критически важное приложение в Kubernetes с replicas: 2. Как гарантировать на уровне манифестов, что при наличии двух узлов в кластере планировщик не разместит оба пода на одном и том же узле, создавая единую точку отказа (SPOF)?
<details>

Для решения этой задачи на уровне манифестов Kubernetes используется механизм Pod Anti-Affinity. Ниже представлен детальный разбор реализации и анализ стратегий.

1. Реализация через Pod Anti-Affinity
Чтобы гарантировать распределение, необходимо использовать селекторы меток (labels) и ключ топологии.
yaml
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution: # Жесткое правило
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - critical-app
        topologyKey: "kubernetes.io/hostname"

topologyKey: "kubernetes.io/hostname": Указывает планировщику, что «границей» для запрета совместного размещения является имя узла.
labelSelector: Определяет группу подов (в данном случае само приложение), к которым применяется правило.

Выбор зависит от приоритетов: строгая отказоустойчивость или доступность любой ценой.

Strategy: Required (Hard)
Когда выбирать: Когда совместное размещение подов на одном узле недопустимо из-за критических рисков (например, высокая нагрузка на CPU/RAM или строгие требования безопасности).
Чем чревато:
Проблема масштабирования: Если в кластере всего 2 узла, вы не сможете масштабировать приложение до replicas: 3. Третий под навсегда останется в статусе Pending.
Отказ узла: Если один из двух узлов выйдет из строя, Kubernetes не сможет перенести второй под на выживший узел. Приложение останется работать в единственном экземпляре (SLA под угрозой), так как правило запрещает «соседство» подов.

Strategy: Preferred (Soft)
Когда выбирать: В большинстве рабочих сценариев.
Преимущества: Планировщик постарается разнести поды по разным узлам. Однако, если доступен только один узел (например, во время аварии или обновления), он все равно запустит обе реплики на нем.
Риск: В момент дефицита ресурсов вы временно получите SPOF, так как оба пода окажутся на одном хосте.
</details>

Вы настроили шифрование секретов в etcd с помощью EncryptionConfiguration. Гарантирует ли это полную безопасность секретов? Какие еще векторы атак могут скомпрометировать секреты в Kubernetes, и как от них защититься?

<details>
    <summary>Ответ</summary>

Шифрование секретов в etcd через EncryptionConfiguration — это отличный шаг, который закрывает риск «кражи жесткого диска» или доступа к бэкапам базы данных. Однако это не гарантирует полную безопасность.
Вот основные векторы атак, которые остаются открытыми, и способы защиты от них:
1. Доступ через Kubernetes API (RBAC)
Шифрование в etcd прозрачно для API-сервера. Если у пользователя или ServiceAccount есть права get или list на секреты, он получит их в открытом (Base64) виде, так как API-сервер сам расшифрует их при выдаче.
Как защититься:
Применяйте принцип Least Privilege (наименьших привилегий).
Используйте инструменты аудита RBAC (например, krane или rbac-lookup), чтобы найти лишние права.
Используйте Admission Controllers для блокировки подозрительных запросов.
2. Секреты в переменных окружения (Environment Variables)
Часто секреты прокидывают в контейнеры через env. Это крайне небезопасно:
Секреты видны в выводе kubectl describe pod.
Любой процесс внутри контейнера (или дочерний процесс) может прочитать их.
Они могут попасть в логи системы мониторинга или логи ошибок.
Как защититься:
Монтируйте секреты как ReadOnly-файлы (Volumes). Это сложнее перехватить через логи.
3. Компрометация узла (Node)
Если злоумышленник получил доступ к уровню ОС на рабочей ноде (Worker Node), он может:
Прочитать секреты из памяти процессов.
Достать секреты из файловой системы /var/lib/kubelet/pods/..., если они смонтированы как тома.
Перехватить токены ServiceAccount, которые лежат в каждом поде.
Как защититься:
Используйте Ephemeral Containers для отладки вместо установки утилит в рабочие образы.
Настройте Seccomp, AppArmor или SELinux профили.
Используйте Runtime Security инструменты (например, Falco), чтобы ловить аномальную активность.
4. Жизненный цикл ключа шифрования (Key Management)
Если ключ шифрования для EncryptionConfiguration хранится локально на мастер-ноде в конфигурационном файле, то взлом мастера ведет к компрометации всех секретов.
Как защититься:
Используйте KMS-плагины (Key Management Service). В этом случае API-сервер не хранит ключ, а обращается к внешнему HSM или облачному хранилищу (Vault, AWS KMS) для расшифровки данных.

</details>

Где Helm хранить данные о релизах (состояниях) и что произойдет если эти данные удалить вручную?
<details>
    <summary>Ответ</summary>
В Helm 3 данные о релизах хранятся непосредственно в кластере Kubernetes, а не на стороне клиента или в отдельном сервисе (как это было в Helm 2 с Tiller).
Где именно хранятся данные
По умолчанию Helm использует Kubernetes Secrets. 

Местоположение: В том же namespace, куда был установлен релиз.
Формат имени: Секреты имеют названия вида sh.helm.release.v1.[RELEASE_NAME].v[REVISION], где v1, v2 и т.д. — это номер ревизии релиза.
Содержимое: Внутри секрета находится сжатый (gzip) и закодированный (base64) объект, содержащий метаданные, значения (values) и сгенерированные манифесты ресурсов. 

(Примечание: С помощью переменной HELM_DRIVER можно сменить хранилище на ConfigMaps или внешнюю базу SQL, но это редко используется на практике) 

Что произойдет, если удалить эти данные вручную
Если вы удалите секреты релиза через kubectl delete secret, последствия будут следующими:
Потеря управления через Helm: Helm «забудет» о существовании релиза. Команды helm list, helm history или helm upgrade больше не увидят установленное приложение.
Ресурсы останутся в кластере: Удаление секретов Helm не удаляет сами поды, сервисы или деплойменты в Kubernetes. Они продолжат работать как «осиротевшие» (orphaned) объекты.
Невозможность отката: Вы потеряете возможность выполнить helm rollback, так как история состояний (предыдущие ревизии) будет уничтожена.
Конфликты при переустановке: Если вы попытаетесь снова выполнить helm install с тем же именем, возникнет ошибка, так как ресурсы (например, Service) уже существуют в кластере, но не числятся за новым релизом. 

</details>

Работали с HPA/VPA?

Допустим в манифесте HPA установлены параметры minReplicas: 2, maxReplicas: 15, а в деплойменте replicas: 10. HPA держит 2 пода, так как нагрузки нет, что произойдет при обновлении?

<details>
    <summary>Ответ</summary>
Произойдет кратковременный всплеск количества подов до 10 (плюс maxSurge), после чего HPA вернет их число к 2.

Применение манифеста: Когда вы запускаете kubectl apply, контроллер Deployment видит в манифесте replicas: 10 и начинает обновлять ReplicaSet, ориентируясь на это число.
Процесс Rollout: В зависимости от стратегии (например, RollingUpdate), Kubernetes создаст новые поды версии 2, пока их общее количество не достигнет целевых 10 (с учетом разрешенного превышения maxSurge).
Вмешательство HPA: Контроллер HPA работает в цикле (обычно каждые 15 секунд). Как только наступит следующая итерация проверки, HPA увидит, что текущая нагрузка по-прежнему низкая и требует всего 2 подов.
Масштабирование вниз: HPA перезапишет поле replicas в Deployment обратно на 2. После этого лишние поды будут удалены.

На короткий промежуток времени может возникнуть всплеск («дрожание»):
Если планировщик и Kubelet успеют сработать быстро, вы увидите, как поды начнут переходить в статус Pending или ContainerCreating (вплоть до 10 штук).
Затем HPA пришлет команду на масштабирование вниз, и лишние поды будут удалены (Terminating).
</details>

Что такое сигнал и в чем разница между SIGKILL и SIGTERM?

<details>
  <summary>Ответ</summary>

Сигнал - в Unix-like операционных системах - асинхронное (в случайное время) уведомление процесса для обработки какого-либо события. Один из основных способов взаимодействия между процессами.

SIGINT — это сигнал, используемый для корректного завершения процесса, обычно инициируемый пользователем. SIGTERM — это сигнал, используемый для корректного завершения процесса, который может быть отправлен другими процессами или самой системой. SIGKILL — это сигнал, используемый для принудительного завершения процесса без выполнения каких-либо операций по очистке. Понимание различий между этими сигналами важно для эффективного управления процессами в системе Linux.

Сначала всегда SIGTERM (команда kill <PID> по умолчанию шлет именно его). Дайте программе 5-10 секунд, чтобы она успела корректно «закрыться».

SIGKILL (kill -9 <PID>) — это «последний аргумент». Используйте его только если процесс завис намертво и не реагирует на обычный kill. 

Важный нюанс: SIGKILL не поможет, если процесс находится в состоянии Uninterruptible sleep (D) (например, ждет ответа от зависшего сетевого диска или «железа»). В этом случае ядро не сможет убить процесс до тех пор, пока ресурс не ответит.

</details>

Как определяется порядок источников для разрешения имен в Linux?

<details>
  <summary>Ответ</summary>

В модели OSI протокол DNS работает на 7-м уровне — Прикладном (Application Layer). 
Для передачи данных DNS чаще всего использует протокол UDP (порт 53) из-за его скорости, но переключается на TCP, если объем передаваемой информации превышает 512 байт.

Порядок источников для разрешения имен (hostnames) в Linux определяется в файле конфигурации /etc/nsswitch.conf в строке hosts:. По умолчанию (обычно hosts: files dns) система сначала проверяет локальный файл /etc/hosts (files), а затем DNS-серверы (dns), указанные в /etc/resolv.conf

</details>

<details>
  <summary>Ответ</summary>

Да, в Ansible идемпотентность является фундаментальным принципом. Это означает, что повторный запуск плейбука (playbook) не изменит систему, если она уже находится в нужном состоянии. Однако идемпотентность нарушается при использовании командных модулей (shell, command, raw) и при неправильном использовании директив, требующих явного указания changed_when: false.

## Доп. вопросы

Что такое дрифт конфигурации? И как бы вы с этим боролись?

<details>
  <summary>Ответ</summary>

Борьба с «дрифтом» (расхождением реальности и кода) — это не только вопрос инструментов, но и вопрос культуры доступа. В крупных проектах вроде ВСК или облачных инфраструктур это решается на трех уровнях:

1. Технический уровень (Drift Detection)
Если используется Terraform/OpenTofu:
Периодические проверки (Check Pipelines): В GitLab CI настраивается расписание (Schedule), которое запускает terraform plan -detailed-exitcode. Если стейт не совпадает с реальностью, пайплайн падает и шлет алерт в Slack/Telegram.
Terraform Cloud / Atlantis: Использование инструментов, которые блокируют внесение изменений вне Pull Request. Atlantis позволяет видеть результат plan прямо в комментариях к MR.
Если используется Kubernetes/OpenShift:
GitOps (ArgoCD / Flux): Это «серебряная пуля» от дрифта в K8s. ArgoCD постоянно сравнивает Git-репозиторий с живым кластером.
Если кто-то поправил replicas или image руками через kubectl edit, ArgoCD подсветит статус OutOfSync.
Self-healing: Можно включить режим Prune и SelfHeal, тогда ArgoCD автоматически откатит ручные правки к состоянию в Git через несколько секунд.

1. Уровень прав доступа (RBAC)
Самый эффективный способ борьбы с ручными правками — забрать права на запись.
Принцип Read-Only: У админов и разработчиков в prod-контурах права только на просмотр (get, list, watch).
Service Accounts: Права на edit/patch есть только у сервисного аккаунта GitLab CI или ArgoCD.
Break-glass access: Ручной доступ разрешается только в случае инцидента через временные токены с логированием всех действий (Audit Log).

1. Уровень политик (Policy as Code)
Использование Kyverno или OPA Gatekeeper:
Можно запретить изменение определенных аннотаций или лейблов, которые важны для инфраструктуры.
Запрет на деплой ресурсов, у которых нет обязательного лейбла managed-by: terraform.

1. Организационный уровень
Immutable Infrastructure: Пропаганда идеи, что любой инстанс или под может быть удален в любой момент. Если изменение внесено руками, оно исчезнет при первом же рестарте или обновлении пайплайна.
Infrastructure drift as a Bug: Фиксация каждого случая дрифта как инцидента. Выяснение причин: «Почему коллеге пришлось зайти руками?». Возможно, пайплайн слишком долгий или Terraform-модуль неудобный.

</details
